## Build for production

```bash
npm run build
```

This generates an optimized production build in the `.next/` directory. All documentation pages are statically pre-rendered for fast loading.

## Environment variables

Before deploying, set the `THALLY_SITE_URL` environment variable to your production domain. This is used for OpenGraph metadata and canonical URLs:

```bash
THALLY_SITE_URL=https://docs.yourproduct.com
```

Depending on which features you use, you may also want to set:

| Variable | Enables |
| --- | --- |
| `THALLY_SITE_URL` | OpenGraph metadata and canonical URLs (recommended for every deploy) |
| `THALLY_CLOUD_SITE_TOKEN` | Connects this deployment to its Thally Cloud site (server-only) |
| `THALLY_CLOUD_URL` | Overrides the Thally Cloud control-plane URL for staging or a public development tunnel |
| `THALLY_ADMIN_PASSWORD` | Break-glass Owner access to the `/admin` dashboard |
| `THALLY_OIDC_ISSUER` / `THALLY_OIDC_CLIENT_ID` / `THALLY_OIDC_CLIENT_SECRET` | Team sign-in to the dashboard via your IdP |
| `THALLY_AUTH_SECRET` | Signs admin sessions and encrypts stored secrets (required with OIDC) |
| `THALLY_DATABASE_URL` / `THALLY_DATABASE_TOKEN` | Durable admin storage (settings, rate limits) on serverless hosts |

Only `THALLY_SITE_URL` is needed for a basic docs site. Admin storage works with zero config locally; on serverless you point Thally at a remote database (below). See [Admin dashboard & team access](/guides/admin-dashboard) for the auth variables in context.

## Connect a self-hosted site to Thally Cloud

After linking the repository in Thally Cloud, copy the one-time site credential and add it to your hosting provider as:

```bash
THALLY_CLOUD_SITE_TOKEN=thally_site_...
```

Keep it server-side: do not commit it, put it in `docs.json`, or rename it with a `NEXT_PUBLIC_` prefix. Redeploy or restart the site, then visit any documentation page. The runtime calls its own server-only handshake route, exchanges the credential for a short-lived entitlement grant, and reports the deployment URL to Thally Cloud automatically. It reports `THALLY_SITE_URL` when configured; otherwise it uses the origin that served the page. This behavior is provider-independent and requires no Vercel, Netlify, or other hosting API.

Production sites use `https://app.thally.io` by default. For local Thally Cloud development, a deployed docs site cannot reach `localhost` on your laptop. Expose Thally Cloud through a secure public tunnel and set its URL on the docs deployment instead:

```bash
THALLY_CLOUD_URL=https://your-cloud-tunnel.example
```

The token never enters browser JavaScript, and Thally Cloud does not need access to your hosting provider's environment variables.

<Note type="info" title="Thally Cloud services">
  AI chat answers, [Track](/guides/ai-features#free-vs-thally-cloud), and traffic analytics are Thally Cloud services. They carry their own variables (for example `ANTHROPIC_API_KEY` and `THALLY_ANALYTICS_DB_URL`) that light up when a deployment connects Thally Cloud; in the free, self-hosted template their admin panels stay locked and none are needed to ship your docs.
</Note>

## Durable admin storage

The free admin dashboard keeps its runtime state (settings overrides, rate-limit counters) in **libSQL** (SQLite-compatible). Locally this is automatic — no account, no setup.

| Environment | What happens |
| --- | --- |
| **Local (`npm run dev`)** | Admin state writes to `.data/thally.db`, created on first use. |
| **Server with a persistent disk** (Docker volume, VPS) | The same on-disk file works if the volume survives restarts. |
| **Serverless** (Vercel, Lambda, etc.) | The filesystem is ephemeral. Without a remote database, runtime admin settings may not persist across deploys. |

Agent Readiness still works without a database — it is computed from your docs content directly.

### Set up Turso (recommended for Vercel)

1. Create a free database (CLI or [Turso dashboard](https://turso.tech)):

```bash
brew install tursodatabase/tap/turso   # or follow Turso's install docs
turso auth login
turso db create your-docs
turso db show your-docs --url
turso db tokens create your-docs
```

2. Add these environment variables on your host (Production):

```bash
THALLY_DATABASE_URL=libsql://your-docs-….turso.io
THALLY_DATABASE_TOKEN=…
```

3. Redeploy. Thally creates the required tables on first connect — you do not run migrations by hand.

<Note type="info" title="Local data stays local">
  Your laptop's `.data/` files are not uploaded to production. Runtime settings you make in the deployed dashboard are stored in the remote database.
</Note>

## Vercel

The fastest way to deploy — push to GitHub and import the repo:

1. Push your docs repo to GitHub
2. Go to [vercel.com/new](https://vercel.com/new) and import the repository
3. Set `THALLY_SITE_URL` in the environment variables
4. For durable admin settings on serverless, also set the Turso variables from [Durable admin storage](#durable-admin-storage)
5. Deploy

Vercel auto-detects Next.js and configures everything.

## Netlify

```bash
npm run build
```

Set the build command to `npm run build` and the publish directory to `.next`. You may need the `@netlify/plugin-nextjs` plugin for full Next.js support.

## Docker

Create a `Dockerfile`:

```dockerfile
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:18-alpine AS runner
WORKDIR /app
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
COPY --from=builder /app/public ./public
COPY --from=builder /app/src/content ./src/content
COPY --from=builder /app/openapi.yaml ./
COPY --from=builder /app/docs.json ./

EXPOSE 3000
CMD ["npm", "start"]
```

Build and run:

```bash
docker build -t my-docs .
docker run -p 3000:3000 my-docs
```

## Cloudflare Workers

Thally ships an OpenNext configuration for Cloudflare Workers. Workers preserve
the server routes used by search, agent endpoints, authentication, analytics,
and the Try It proxy; a static Pages export does not.

```bash
npm run preview:cloudflare  # build and test locally in workerd
npm run deploy:cloudflare   # build and deploy with Wrangler
```

Authenticate Wrangler with `npx wrangler login` for local deployment, or set a
scoped `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` in CI. Keep those
values in the deployment environment—never in `docs.json` or the repository.

## Static export

If you don't need server-side features (like the Try It proxy), you can export as a fully static site. Add to `next.config.ts`:

```ts
const nextConfig = {
  output: 'export',
}
```

Then run `npm run build` — the output will be in the `out/` directory, ready for any static host.

<Note type="warning" title="Static export limitation">
  The API "Try It" panel requires a server-side proxy route and won't work with static export. All other features work fine.
</Note>

## SEO and feeds

Once deployed, Thally automatically serves:

- **`/sitemap.xml`** — includes all doc pages, API reference pages, and the changelog
- **`/robots.txt`** — allows crawlers on all pages, blocks internal API routes (`/api/og`, `/api/try-it`)
- **`/changelog/rss.xml`** — RSS feed for changelog updates
- **Dynamic OG images** — every page gets a branded social preview image via `/api/og`

All of these use `THALLY_SITE_URL` to generate absolute URLs, so make sure it's set before deploying.
